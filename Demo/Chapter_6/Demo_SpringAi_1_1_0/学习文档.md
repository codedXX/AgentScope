## 1. 先搞懂几个核心概念

在看代码之前，先用大白话把几个名词讲清楚，否则代码会很难懂。

### 1.1 什么是大模型（ChatModel）？
就是 ChatGPT、通义千问（qwen）、DeepSeek 这类"会聊天的 AI"。在代码里，它被抽象成一个对象叫 `ChatModel`。你给它一句话，它返回一段回答。

### 1.2 什么是 Agent（智能体）？
**普通调用大模型** = 你问一句，它答一句，仅此而已。

**Agent（智能体）** = 给大模型装上了"手脚"和"大脑决策能力"，让它能**自己思考、自己决定要不要调用工具、自己执行任务**。

打个比方：
- 普通大模型像一个**只能动嘴的人**（只会回答问题）。
- Agent 像一个**有手有脚、会自己做计划的人**（能查天气、查数据库、调 API，还能把复杂任务拆成几步来做）。

### 1.3 什么是 ReAct？
ReAct = **Reasoning（推理）+ Acting（行动）**。

这是 Agent 的一种工作模式，循环执行：
```
思考（我需要什么信息？）→ 行动（调用工具拿信息）→ 观察（看工具返回了啥）→ 再思考 → ...直到能回答用户
```
本项目里的 `ReactAgent` 就是这种模式的 Agent，是**最基础的、能干活的单个智能体**。

### 1.4 什么是工作流 Agent（FlowAgent）？
当一个任务很复杂，需要**多个 Agent 配合**完成时，就需要把它们"组织"起来。组织的方式有几种：

| 工作流 Agent 类型 | 作用（大白话） |
|---|---|
| `SequentialAgent` | **顺序执行**：先让 A 干，A 干完让 B 干（像流水线）|
| `ParallelAgent` | **并行执行**：让 A 和 B 同时干，互不依赖（像两个人分头办事）|
| `SupervisorAgent` | **主管/路由**：一个领导决定把任务派给谁去做 |
| `LlmRoutingAgent` | **智能路由**：让大模型来判断该派给谁（比上面更智能）|

> **关键区别**：`ReactAgent` 是**真正干活的人**；工作流 Agent（Sequential/Parallel/Supervisor）是**组织者/管理者**，它们本身不干活，只负责把干活的 Agent 按某种逻辑串起来。

### 1.5 什么是 DashScope？
DashScope（灵积）是**阿里云提供的大模型服务平台**。通义千问 qwen、DeepSeek 等模型都通过它来调用。你需要一个 `api-key`（密钥）才能用，调用是要花钱的（按 token 计费）。

## 2. 项目整体结构

```
Demo_SpringAi_1_1_0/
├── pom.xml                          # Maven 依赖配置（项目需要哪些库）
└── src/main/
    ├── java/com/imooc/
    │   ├── SpringAiApplication.java          # 程序启动入口
    │   ├── controller/
    │   │   ├── ReActAgentController.java     # 【场景一】单个 Agent 流式对话
    │   │   └── MultiAgentsController.java    # 【场景二】多 Agent 协同工作流
    │   └── tools/
    │       └── WeatherTool.java              # 自定义工具（天气查询，本项目里是空壳示例）
    └── resources/
        ├── application.yml                   # 配置文件（端口、密钥等）
        └── logback-spring.xml                # 日志配置
```

整个项目的"运行逻辑"非常简单：
1. 启动 Spring Boot 程序（监听 8080 端口）。
2. 浏览器访问某个网址（如 `http://localhost:8080/simple/agent`）。
3. 对应的 Controller 方法被触发，创建 Agent → 调用大模型 → 在**控制台（后台）打印结果**。

> ⚠️ 注意：本项目的结果是 `System.out.println` 打印在 **IDEA 控制台**里的，不是返回给浏览器的。访问网址只是为了"触发"代码执行。

## 3. 项目依赖（pom.xml）讲解

`pom.xml` 是 Maven 的配置文件，告诉项目"我需要用到哪些第三方库"。

```xml
<properties>
    <!-- Spring AI Alibaba 的版本号 -->
    <spring-ai-alibaba.version>1.1.0.0-RC2</spring-ai-alibaba.version>
    <spring-ai-alibaba-extensions.version>1.1.0.0-RC2</spring-ai-alibaba-extensions.version>
    <springboot.version>3.4.5</springboot.version>

    <maven.compiler.source>17</maven.compiler.source>   <!-- 用 JDK 17 编译 -->
    <maven.compiler.target>17</maven.compiler.target>
</properties>
```
**讲解**：这里定义了几个"版本变量"，方便后面统一引用。注意本项目**要求 JDK 17**，你的电脑必须装了 JDK 17 或以上。

```xml
<dependencyManagement>
    <dependencies>
        <!-- BOM：统一管理一堆相关依赖的版本，避免版本冲突 -->
        <dependency>
            <groupId>com.alibaba.cloud.ai</groupId>
            <artifactId>spring-ai-alibaba-bom</artifactId>
            ...
        </dependency>
        ...
    </dependencies>
</dependencyManagement>
```
**讲解**：`dependencyManagement` 里的 BOM（Bill of Materials，物料清单）是一种"版本管理工具"。它只**声明版本**，不真正引入库。好处是：下面真正引依赖时不用写版本号，由 BOM 统一管，避免一堆库版本对不上的问题。

```xml
<dependencies>
    <!-- ① Agent 框架核心：ReactAgent、SequentialAgent 等都来自这里 -->
    <dependency>
        <groupId>com.alibaba.cloud.ai</groupId>
        <artifactId>spring-ai-alibaba-agent-framework</artifactId>
    </dependency>

    <!-- ② DashScope 启动器：让你能调用阿里云的大模型（qwen、deepseek 等）-->
    <dependency>
        <groupId>com.alibaba.cloud.ai</groupId>
        <artifactId>spring-ai-alibaba-starter-dashscope</artifactId>
    </dependency>

    <!-- ③ Web 启动器：提供 @RestController、@GetMapping 等能力（必须加）-->
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-web</artifactId>
    </dependency>
</dependencies>
```
**讲解**：真正用到的就这 3 个依赖：
- **①** 是核心，提供所有 Agent 相关的类。
- **②** 是让你能连上阿里云大模型。
- **③** 是 Web 支持，让你能通过浏览器访问网址来触发代码。注释也强调了"必须添加"。

```java
@SpringBootApplication          // 标记这是一个 Spring Boot 应用
```

## 5. 场景一：单个 Agent 流式对话（ReActAgentController）

文件：`controller/ReActAgentController.java`

这是**最基础的例子**：创建一个 Agent，问它一个问题，让它**流式地**（一个字一个字地）把答案打印出来。

### 5.1 接口入口

```java
@RestController     // 标记这是一个处理网络请求的控制器
public class ReActAgentController {

    @GetMapping(value = "/simple/agent", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public void streamChat() throws GraphRunnerException {
```
**讲解**：
- `@GetMapping("/simple/agent")`：当浏览器访问 `http://localhost:8080/simple/agent` 时，执行下面这个方法。
- `produces = MediaType.TEXT_EVENT_STREAM_VALUE`：声明返回类型是 **SSE（Server-Sent Events，服务器推送流）**，适合"流式输出"的场景。
- 返回值是 `void`（无返回），因为结果是打印在控制台的，不返回给浏览器。

### 5.2 第一步：创建大模型连接

```java
// 创建 DashScope API 实例（连接阿里云）
DashScopeApi dashScopeApi = DashScopeApi.builder()
        .apiKey("sk-255506ca196b48f38e686b3e82efac58")   // 你的 API 密钥
        .build();

// 创建 ChatModel（指定用哪个大模型）
ChatModel chatModel = DashScopeChatModel.builder()
        .dashScopeApi(dashScopeApi)
        .defaultOptions(
                DashScopeChatOptions.builder()
                        .model("deepseek-r1").build()    // 使用 deepseek-r1 模型
        )
        .build();
```
**讲解**：这是固定套路，分两步：
1. **`DashScopeApi`**：用你的 `apiKey` 建立和阿里云服务器的连接（钥匙）。
2. **`ChatModel`**：在连接的基础上，指定要用哪个具体模型（这里是 `deepseek-r1`，一个擅长推理的模型）。

> ⚠️ **安全提醒**：代码里把 `apiKey` 硬编码（直接写死）了，这是**不推荐的做法**！密钥泄露后别人可以盗用你的额度。正确做法是放到配置文件或环境变量里读取。详见[第 10 节](#10-新手常见疑问--注意事项)。

### 5.3 第二步：准备用户问题和工具

```java
// 用户输入（要问 AI 的问题）
String message = "在咖啡馆里，我也想要杯星巴克，这里的星巴克指的是什么?";

// 定义一个"工具"（这里是天气查询工具）
ToolCallback weatherTool = FunctionToolCallback
        .builder("getWeather", new WeatherTool())   // 工具名 + 工具实现类
        .description("查询城市天气")                  // 告诉 AI 这个工具能干啥
        .inputType(String.class)                     // 工具的输入类型是字符串
        .build();
```
**讲解**：
- `message` 是要问 AI 的问题。
- `weatherTool` 是一个**工具**。所谓工具，就是"让 AI 能调用的一个 Java 方法"。这里把 `WeatherTool` 这个类包装成了 Agent 能用的工具。
  - `.description("查询城市天气")` 很重要：**AI 是靠这段描述来判断"什么时候该调用这个工具"的**。
- ⚠️ 注意：虽然这里定义了 `weatherTool`，但在下面创建 Agent 时这一行 `.tools(weatherTool)` 是**被注释掉的**，所以本例实际上没用到这个工具。这是作者留给你做练习/演示的。

### 5.4 第三步：创建 Agent

```java
ReactAgent agent = ReactAgent.builder()
        .name("weather_agent")                       // Agent 的名字（必须）
        .model(chatModel)                            // Agent 的"大脑"（大模型）
        //.tools(weatherTool)                        // Agent 的"手脚"（工具）—— 这里注释掉了
        //.hooks()                                   // Agent 的"眼睛耳朵"（RAG 知识检索）
        .systemPrompt("You are a helpful assistant") // 系统提示词（给 AI 设定角色）
        //.saver(new MemorySaver())                  // Agent 的"记忆"（多轮对话）—— 注释掉了
        .build();
```
**讲解**：这是用**建造者模式（builder）**一步步"组装"一个 Agent。源码里的注释非常形象地把 Agent 比作一个人：
| 配置项 | 比喻 | 作用 |
|---|---|---|
| `.name(...)` | 名字 | Agent 的标识（必填）|
| `.model(...)` | 大脑 | 用哪个大模型来思考 |
| `.tools(...)` | 手脚 | 能调用哪些工具（本例注释掉了）|
| `.hooks(...)` | 眼睛耳朵 | RAG 检索外部知识（本例注释掉了）|
| `.systemPrompt(...)` | 人设 | 给 AI 设定身份/规则 |
| `.saver(...)` | 记忆 | 记住多轮对话历史（本例注释掉了）|

**所以本例其实只用了"大脑 + 人设"，是最精简的 Agent。**

### 5.5 第四步：流式获取回答

```java
// （注释掉的）阻塞式调用：一次性返回完整答案
// AssistantMessage assistantMessage = agent.call(message);
// assistantMessage.getText();

// 流式调用：返回一个"数据流"，答案会一段段地推送过来
Flux<NodeOutput> stream = agent.stream(message);
```
**讲解**：调用 Agent 有两种方式：
- **`agent.call(message)`（阻塞式）**：等 AI 把整段话想完，一次性返回。简单，但用户要等。
- **`agent.stream(message)`（流式）**：AI 想一个字就推一个字，像 ChatGPT 网页那样逐字蹦出来。本例用的是这种。`Flux` 是响应式编程里表示"数据流"的类型（来自 Reactor 库）。

```java
stream.subscribe(
    output -> {   // 【每收到一段数据时执行】
        if (output instanceof StreamingOutput streamingOutput) {
            OutputType type = streamingOutput.getOutputType();   // 这段数据是什么类型？

            // ① 模型正在逐字输出
            if (type == OutputType.AGENT_MODEL_STREAMING) {
                System.out.print(streamingOutput.message().getText());  // 逐字打印
            }
            // ② 模型输出完毕
            else if (type == OutputType.AGENT_MODEL_FINISHED) {
                System.out.println("\n模型输出完成");
            }

            // ③ 工具调用完成
            if (type == OutputType.AGENT_TOOL_FINISHED) {
                System.out.println("工具调用完成: " + output.node());
            }
            // ④ Hook（钩子）执行完成
            if (type == OutputType.AGENT_HOOK_FINISHED) {
                System.out.println("Hook 执行完成: " + output.node());
            }
        }
    },
    error -> System.err.println("错误: " + error),   // 【出错时执行】
    () -> System.out.println("Agent 执行完成")        // 【全部完成时执行】
);
```
**讲解**：`stream.subscribe(...)` 是"订阅"这个数据流。它有 3 个参数（3 个回调函数）：
1. **第一个参数**：每收到一段数据就执行。代码里通过 `OutputType` 判断数据是什么阶段：
   - `AGENT_MODEL_STREAMING`：模型正在逐字吐答案 → 用 `print`（不换行）逐字拼出来。
   - `AGENT_MODEL_FINISHED`：模型说完了。
   - `AGENT_TOOL_FINISHED`：某个工具调用完了。
   - `AGENT_HOOK_FINISHED`：某个钩子执行完了。
2. **第二个参数**：发生错误时执行（打印错误）。
3. **第三个参数**：整个流结束时执行（打印"Agent 执行完成"）。

> 💡 `output instanceof StreamingOutput streamingOutput` 是 Java 16+ 的语法（模式匹配），意思是"如果 output 是 StreamingOutput 类型，就自动转换并赋值给变量 streamingOutput"。

**小结场景一**：连接大模型 → 组装一个最简 Agent → 流式提问 → 逐字打印答案。这是理解 Agent 的基础。

## 6. 场景二：多 Agent 协同工作流（MultiAgentsController）

文件：`controller/MultiAgentsController.java`

这是本项目的**重头戏**：用多个 Agent 配合，完成一个"旅游规划"的复杂任务。

### 6.1 业务场景（先理解要干什么）

需求：用户说"帮我规划深圳到惠州的驾车路线 + 制定 1 天 1 夜的旅游行程"。

设计思路（源码注释里写得很清楚）：
1. **费用统筹 Agent**（`budgetAgent`）：负责算钱，被嵌入到其他 Agent 里当工具用。
2. **路线制定 Agent**（`routeMakingAgent`）：负责规划驾车路线。
3. **行程规划 Agent**（`tripPlannerAgent`）：负责规划景点、住宿、美食。
4. 路线制定和行程规划**互不依赖，可以并行**。
5. **主管 Agent**（`managerAgent`）：负责把任务分发给合适的 Agent。
6. 最后用工作流把它们**按顺序 + 并行**组织起来执行。

用一张图表示最终的组织结构：

```
SequentialAgent（顺序工作流：先路由，再并行）
   │
   ├──① SupervisorAgent（主管：决定任务派给谁）
   │       ├── tripPlannerAgent（行程规划）
   │       └── routeMakingAgent（路线制定）
   │
   └──② ParallelAgent（并行工作流：两个同时干）
           ├── tripPlannerAgent ── 内部用 budgetAgent（算钱）当工具
           └── routeMakingAgent ── 内部用 budgetAgent（算钱）当工具
```

### 6.2 创建大模型（和场景一类似）

```java
DashScopeApi dashScopeApi = DashScopeApi.builder()
        .apiKey("sk-255506ca196b48f38e686b3e82efac58")
        .build();

ChatModel chatModel = DashScopeChatModel.builder()
        .dashScopeApi(dashScopeApi)
        .defaultOptions(
                DashScopeChatOptions.builder()
                        .model("qwen3-32b").build()   // 这里用通义千问 qwen3-32b
        )
        .build();
```
**讲解**：和场景一一样，只是模型换成了 `qwen3-32b`。注意这个 `chatModel` 会被下面所有 Agent 共用。

### 6.3 创建"费用统筹 Agent"（最底层的工具 Agent）

```java
ReactAgent budgetAgent = ReactAgent.builder()
        .name("budgetAgent")
        .description("你是负责预估费用")   // 描述它的职责
        .model(chatModel)
        .tools()                          // 它自己不带额外工具
        .build();
```
**讲解**：这是一个负责"算钱"的 Agent。它本身是个 `ReactAgent`，但**关键在于**：下面会把它当成"工具"塞进别的 Agent 里。这体现了一个重要思想——**一个 Agent 可以作为另一个 Agent 的工具**。

### 6.4 创建"路线制定 Agent"

```java
ReactAgent routeMakingAgent = ReactAgent.builder()
        .name("routeMakingAgent")
        .description("负责制定驾车路线")
        .instruction(                     // instruction：更详细的工作指令
                "擅长驾车路线的规划," +
                "包括了路线距离,高速费用,油费" +
                "返回性价比最优的驾车路线, 及预估费用," +
                "以表格形式展示"
        )
        .model(chatModel)
        // 把 budgetAgent 当作工具塞进来！（一个 Agent 作为另一个 Agent 的工具）
        .tools(AgentTool.getFunctionToolCallback(budgetAgent))
        .returnReasoningContents(true)    // 返回推理过程（思考链）
        .outputKey("route_making")        // 它的输出结果存到名为 route_making 的"格子"里
        .build();
```
**讲解**：重点理解 3 个配置：
- **`.instruction(...)`**：比 `description` 更详细的"工作说明书"，告诉这个 Agent 具体该怎么干活、输出什么格式。
- **`.tools(AgentTool.getFunctionToolCallback(budgetAgent))`**：把 `budgetAgent` 转换成工具塞进来。意思是"路线 Agent 在规划路线时，如果需要算费用，就调用费用 Agent"。
- **`.outputKey("route_making")`**：给它的输出结果起个"键名"。后面取结果时用 `state.value("route_making")` 就能拿到它的输出。这就像把结果存进一个有标签的盒子。

### 6.5 创建"行程规划 Agent"

```java
ReactAgent tripPlannerAgent = ReactAgent.builder()
        .name("tripPlanAgent")
        .description("负责规划旅游行程")
        .instruction(
                "擅长旅游行程的规划," +
                "包括了景点,本地小吃,住宿" +
                "返回性价比最优的行程规划, 及预估费用," +
                "以表格形式展示"
        )
        .model(chatModel)
        .tools(AgentTool.getFunctionToolCallback(budgetAgent))  // 同样把费用 Agent 当工具
        .outputKey("trip_plan")                                 // 输出存到 trip_plan
        .build();
```
**讲解**：和路线 Agent 几乎一样，只是职责变成"规划景点/美食/住宿"，输出键名是 `trip_plan`。它同样把 `budgetAgent` 当工具来算钱。

### 6.6 创建"并行 Agent"（让路线和行程同时干）

```java
ParallelAgent parallelAgent = ParallelAgent.builder()
        .name("parallelAgent")
        .description("并行执行多个任务, 以表格形式进行总结")
        .subAgents(List.of(tripPlannerAgent, routeMakingAgent))  // 它管理的两个子 Agent
        .mergeOutputKey("merged_results")                        // 合并后的结果存到这里
        .build();
```
**讲解**：
- `ParallelAgent` 是一个**工作流 Agent**，本身不干活，只负责"让旗下的 Agent 同时开工"。
- `.subAgents(...)`：指定它管的两个 Agent——行程规划 和 路线制定。因为这俩没有依赖关系，所以可以并行（同时跑），节省时间。
- `.mergeOutputKey("merged_results")`：把两个 Agent 的输出**合并**后存到 `merged_results` 键里。

### 6.7 创建"主管 Agent"（负责派活）

```java
// 注释掉的方案：用 LlmRoutingAgent（纯靠大模型路由）
// LlmRoutingAgent managerAgent = LlmRoutingAgent.builder()...

// 实际使用：SupervisorAgent（主管模式）
SupervisorAgent managerAgent = SupervisorAgent.builder()
        .name("managerAgent")
        .description("智能路由到合适的Agent去执行")
        .instruction("""
                ## 可用的子Agent及其职责

                ### routeMakingAgent
                - **功能**: 擅长驾车路线制定

                ### tripPlanAgent
                - **功能**: 擅长住宿,景点的行程规划
                """)
        .model(chatModel)
        .subAgents(List.of(tripPlannerAgent, routeMakingAgent))
        .build();
```
**讲解**：
- `SupervisorAgent`（主管）是一个**工作流 Agent**，扮演"领导"角色。它根据用户的需求，**决定把任务派给哪个子 Agent**。
- `.instruction(...)` 用了 Java 的**文本块语法**（三个双引号 `"""`），可以方便地写多行字符串。这段说明书告诉主管："你手下有这两个 Agent，各自擅长什么"，主管据此来分派任务。
- 源码里还注释了另一个方案 `LlmRoutingAgent`——两者都能做"路由分发"，区别在于实现机制。作者保留了注释让你对比学习。

### 6.8 创建"顺序工作流"（把所有东西串起来）

```java
SequentialAgent sequentialAgent = SequentialAgent.builder()
        .name("sequentialAgent")
        .description(
                "旅游规划工作流：" +
                "路由选择合适Agent执行 ->" +
                "并行执行路线制定和行程规划 -> " +
                "以表格形式总结"
        )
        .subAgents(List.of(managerAgent, parallelAgent))   // 先 managerAgent，再 parallelAgent
        .build();
```
**讲解**：
- `SequentialAgent`（顺序工作流）是**最外层的总指挥**。它让旗下的 Agent **按顺序**执行。
- `.subAgents(List.of(managerAgent, parallelAgent))`：注意顺序——**先执行主管 Agent（路由），再执行并行 Agent**。
- 所以整体流程是：主管先分析/路由 → 然后并行跑路线和行程 → 出结果。

### 6.9 启动工作流并取结果

```java
// 启动整个工作流（把用户需求传进去）
Optional<OverAllState> res = sequentialAgent.invoke(
        "帮我规划深圳到惠州性价比高的驾车路线, 以及制定1天1夜的旅游行程"
);

// 取结果
if (res.isPresent()) {
    OverAllState state = res.get();

    System.out.println("======= 驾车路线 ============");
    System.out.println(state.value("route_making"));   // 取路线 Agent 的输出

    System.out.println("======= 旅游行程 ============");
    System.out.println(state.value("trip_plan"));       // 取行程 Agent 的输出

    System.out.println("======= 表格总结 ============");
    System.out.println(state.value("merged_results"));  // 取合并后的总结
} else {
    System.out.println("没有结果: " + res);
}
```
**讲解**：
- **`sequentialAgent.invoke(...)`**：把用户需求丢给最外层工作流，启动整条流水线。`invoke` 是阻塞式的（等全部跑完才返回）。
- **`Optional<OverAllState>`**：`Optional` 表示"结果可能有也可能没有"，用 `res.isPresent()` 判断有没有结果，避免空指针错误。
- **`OverAllState`**：可以理解成一个"共享黑板"或"全局状态仓库"。所有 Agent 的输出都存在这块黑板上，用之前设的 `outputKey` 当标签。
- **`state.value("route_making")`**：用键名从黑板上取出对应 Agent 的输出。这就对应了前面 6.4/6.5/6.6 里设的 `outputKey` 和 `mergeOutputKey`。

**小结场景二**：这就是 "Agentic AI"（多智能体协同）的雏形——把一个复杂任务拆给多个专职 Agent，再用工作流（顺序/并行/主管）把它们组织起来协同完成。

## 10. 新手常见疑问 & 注意事项

### Q1：为什么浏览器访问后页面是空白/一直转圈？
因为这两个接口返回值是 `void`，结果是 `System.out.println` 打到控制台的，**不会返回给浏览器**。请去 IDEA 控制台看输出。另外接口声明了 SSE 流式类型，浏览器可能会保持连接，属于正常现象。

### Q2：API 密钥硬编码安全吗？
**不安全！** 把密钥写死在代码里有两个风险：
1. 代码提交到 Git/GitHub 后密钥就泄露了，别人能盗刷你的额度。
2. 换密钥得改代码、重新编译。

**推荐做法**：把密钥放到 `application.yml` 或环境变量里：
```yaml
spring:
  ai:
    dashscope:
      api-key: ${DASHSCOPE_API_KEY}   # 从环境变量读取
```
然后代码里通过注入的方式拿到，而不是 `new` 出来硬编码。

### Q3：`ReactAgent` 和 `SequentialAgent/ParallelAgent/SupervisorAgent` 到底啥区别？
- **`ReactAgent`** = 干活的"员工"，真正调用大模型、用工具完成具体任务。
- **`Sequential/Parallel/Supervisor Agent`** = "管理者/组织者"，自己不干活，只负责把员工按某种逻辑（顺序/并行/分派）组织起来。

### Q4：`outputKey` / `mergeOutputKey` / `state.value(...)` 是怎么对应的？
它们是一套"存取数据"的机制：
- 创建 Agent 时用 `outputKey("xxx")` 给输出**起名/贴标签**。
- 运行完后，所有输出都存在 `OverAllState`（共享黑板）里。
- 用 `state.value("xxx")` 按标签**取出**对应输出。

### Q5：本项目代码里有哪些"未启用"的部分（注释掉的）值得我去试？
- 场景一里 `.tools(weatherTool)`、`.saver(new MemorySaver())`：取消注释 + 完善 `WeatherTool`，可以体验"工具调用"和"多轮记忆"。
- 场景二里 `LlmRoutingAgent`：可以替换 `SupervisorAgent` 来对比两种路由方式。

### Q6：调用大模型要花钱吗？
**要**。DashScope 按 token（字数）计费。多 Agent 工作流会调用大模型很多次，测试时注意你的账户额度。
