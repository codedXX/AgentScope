## 📌 开始前：先建立 3 个心智模型

看代码前，先把这 3 句话刻在脑子里，不然会看不懂：

### 1. 什么是 Agent（智能体）？
> **Agent = 大模型(大脑) + 工具(手脚) + 循环(干活方式)**

它不是普通程序。普通程序是你写死 `if...else`；Agent 是**大模型自己思考"下一步干什么"，然后调用工具去做**，循环往复直到完成。

### 2. 这个项目在干什么？
> **1 个"主管"指挥 2 个"专家"做旅游规划。**
> - 主管（ManagerAgent）：拆解任务、派活、汇总
> - 专家1（RouteMakingAgent）：算驾车路线
> - 专家2（TripPlannerAgent）：规划景点/美食/住宿

### 3. ReAct 循环（每个 Agent 内部都在做的事）
```
想(Reason) → 做(Act，调工具) → 看结果(Observe) → 再想… → 直到完成
```

✅ **建立好这 3 个概念，再往下看。**

## 第②站：程序入口（从你最熟悉的地方进）

作为新手，**别从 Agent 开始**，从你熟悉的 Spring Boot Controller 开始。

**按顺序看这 4 个文件**：

### 2.1 启动类（最简单）
[ManagerAgentApplication.java](manager_agent/src/main/java/com/imooc/managerAgent/ManagerAgentApplication.java)
```java
@SpringBootApplication(scanBasePackages = {"com.imooc"})
public class ManagerAgentApplication {
    public static void main(String[] args) { SpringApplication.run(...); }
}
```
就是个标准 Spring Boot 启动类，没啥特别。`scanBasePackages = "com.imooc"` 是为了能扫到 `commons` 里的 Bean。

### 2.2 请求/响应的数据格式
- [PromptSchema.java](commons/src/main/java/com/imooc/commons/data/PromptSchema.java)：请求体 `{ "prompt": "..." }`
- [ResponseSchema.java](commons/src/main/java/com/imooc/commons/data/ResponseSchema.java)：响应体 `{ "response": "..." }`

就两个字段，很简单。

### 2.3 REST 入口（整个项目的起点）
[ManagerAgentController.java](manager_agent/src/main/java/com/imooc/managerAgent/controller/ManagerAgentController.java)
```java
@RequestMapping(value = "/trip", method = POST)
public ResponseSchema tripPlan(@RequestBody PromptSchema input) {
    ResponseSchema response = managerAgent.run(input.getPrompt());  // ← 核心就这一句
    return response;
}
```

**你要明白的**：
- 用户 `POST /trip` 进来，参数是 `prompt`。
- 真正干活的是 `managerAgent.run(...)`，结果直接返回。
- **整个项目的"主线"就是从这一句 `run()` 开始的。** 记住它，后面会顺着它往下挖。

**✅ 检查点**：你能说清"请求从哪进、走到哪个方法、返回什么"。

## 第③站：Agent 工厂（理解"一个 Agent 怎么造出来"）

所有 Agent 都用同一个工厂方法创建，先看懂它，后面看每个 Agent 就轻松了。

### 3.1 Agent 的统一创建入口 ⭐核心
[AgentUtils.java](commons/src/main/java/com/imooc/commons/utils/AgentUtils.java) 的 `getReActAgentBuilder()`

**重点看这段**：
```java
ReActAgent.builder()
    .name(name)
    .description(description)
    .model(DashScopeChatModel.builder()   // ← 这就是 Agent 的"大脑"
        .apiKey(aliApiKey)                 //    阿里云的 Key
        .modelName(modelName)              //    用哪个千问模型
        .stream(true)
        .enableThinking(true)
        .build())
    .toolExecutionConfig(...)              // 工具超时配置
```

**你要明白的**：
- 每个 Agent 都要配一个**大模型**（这里是阿里云通义千问 DashScope）——这就是它的"大脑"。
- 这个方法返回的是 **`Builder`（半成品）**，不是成品。为什么？因为每个 Agent 还要加自己的"个性"（角色、工具），所以先给个半成品，各 Agent 再往上加，最后自己 `.build()`。

### 3.2 消息的基本单位
[PromptUtils.java](commons/src/main/java/com/imooc/commons/utils/PromptUtils.java)
```java
Msg.builder().role(MsgRole.USER).content(...TextBlock...).build()
```
**你要明白的**：跟大模型对话的基本单位是 `Msg`（消息）。这个工具就是把用户的字符串包装成框架要的 `Msg` 对象。

**✅ 检查点**：你能说出"造一个 Agent 至少要给它什么"（答：大脑=模型，外加名字/描述）。

## 第④站：最简单的专家 Agent（先看简单的）

3 个 Agent 里，**RouteMakingAgent 最简单**，先看它建立信心。

**看这个文件**：[RouteMakingAgent.java](routeMaking_agent/src/main/java/com/imooc/routeMakingAgent/agents/RouteMakingAgent.java)

```java
@Bean
public ReActAgent getRouteMakingAgent() {
    Toolkit toolkit = toolUtils.getToolkit(mcpClient);   // 工具=百度地图(先当黑盒)
    return agentUtils.getReActAgentBuilder("RouteMakingAgent", "...")
        .sysPrompt("你是一个驾车路线制定助手...")          // ← 给它一个"角色/人设"
        .toolkit(toolkit)                                // ← 给它"工具"
        .build();
}
```

**你要明白的**：
- 一个 Agent = **工厂半成品** + **sysPrompt(角色设定)** + **toolkit(工具)** + `.build()`。
- `sysPrompt` 就是告诉大模型"你是谁、该干嘛"。
- `@Bean`：把它交给 Spring 管理（后面 A2A 会用到，这里先不管）。

**先跳过**：`mcp.getBaiduMapMCP()` 那两行 MCP 初始化——**把 MCP 当成"一个外部工具服务"黑盒**就行，第⑤站再说。

**✅ 检查点**：你能照着写出"创建一个 Agent 的固定套路"。

## 第⑤站：工具体系（Agent 的"手脚"）

Agent 靠工具干活。这个项目的工具有**三种来源**，都在一个类里。

**看这个文件**：[ToolUtils.java](commons/src/main/java/com/imooc/commons/utils/ToolUtils.java)

注意它有 **3 个同名 `getToolkit` 方法（重载）**，对应三种工具：

```java
getToolkit(Object tool)         // ① 普通Java方法当工具 (扫描 @Tool 注解)
getToolkit(McpClientWrapper mcp)// ② MCP外部服务当工具
getToolkit(ReActAgent agent)    // ③ 另一个Agent当工具（子Agent）
```

**你要明白的**：这是 AgentScope 的精髓——**任何东西都能变成"工具"**：一个本地方法、一个外部 MCP 服务、甚至另一个 Agent。

**配合看（可略读）**：[BaiduMapMCP.java](routeMaking_agent/src/main/java/com/imooc/routeMakingAgent/mcp/BaiduMapMCP.java)——看一眼 MCP 客户端怎么用 SSE 连接外部服务，细节不用抠。

**✅ 检查点**：你能说出"工具的 3 种来源"，并知道路线专家用的是第②种(MCP)。

## 🌟 第⑥站：主管 Agent（重头戏，调度核心）

前 5 站都是铺垫，这一站才是项目的灵魂。**回到第②站记住的那句 `managerAgent.run()`**，现在顺着它往下挖。

### 6.1 主管怎么造出来的
[ManagerAgent.java](manager_agent/src/main/java/com/imooc/managerAgent/agents/ManagerAgent.java) 的 `getManagerAgent()`
```java
agentUtils.getReActAgentBuilder("ManagerAgent", "...")
    .sysPrompt("你是一个旅游管理主管...先创建计划...派活给Agent...")
    .planNotebook(planNotebook)            // 计划能力(第⑦站讲)
    .hook(new planHook(planNotebook))      // 拦截器(第⑦站讲)
    .toolkit(toolkit)                      // ← 工具 = 两个远程专家
    .build();
```
跟第④站套路一样，只是多了 `planNotebook` 和 `hook`（**先跳过，第⑦站专门讲**）。重点看 `toolkit`——主管的工具是什么？

### 6.2 把"远程专家"变成工具 ⭐关键
[RemoteAgentTool.java](manager_agent/src/main/java/com/imooc/managerAgent/tool/RemoteAgentTool.java)

这个类有两个方法，结构一样，看懂一个就行：
```java
@Tool(description = "擅长制定最优驾车路线")        // ← 模型靠这句描述决定调不调它
public String callRouteMakingAgent(String prompt) {
    A2aAgent agent = A2aAgent.builder()
        .name("RouteMakingAgent")                 // 要调的远程Agent名字
        .agentCardResolver(new NacosAgentCardResolver(...))  // 通过Nacos找到它
        .build();
    Msg response = agent.call(userMsg).block();   // 跨进程调用远程专家
    return response.getTextContent();             // 把专家的回答返回
}
```

**你要明白的（超级重要）**：
- `@Tool(description=...)` 把一个方法变成工具。**主管的大模型靠读这个 description 来决定"这个任务该调哪个工具"**——这就是它"知道用谁"的原因。
- 方法内部用 **A2A 协议**调用跑在另一个进程(8082/8085)的专家 Agent。

### 6.3 主管怎么跑、怎么拿结果
回到 `ManagerAgent.run()`：
```java
Msg reply = agent.call(List.of(promptUtils.getPrompt(prompt))).block(); // 跑完整个流程
result.response = reply.getTextContent();   // 取最终回复
```
`call().block()` 会跑完整个"拆解→派活→汇总"流程（期间真的去调了远程专家），最后拿最终回复。

**✅ 检查点**：你能讲清"主管怎么决定把某个子任务派给哪个专家"（答：靠 @Tool 的 description 语义匹配）。

## 第⑦站：三大高级机制（主流程懂了再看）

主流程通了，现在看那 3 个之前跳过的"高级配置"。

### 7.1 PlanNotebook —— 自动拆解任务
- 看 [TripPlan.java](manager_agent/src/main/java/com/imooc/managerAgent/plan/TripPlan.java)：配置"最多拆 5 个子任务、免人工确认"。
- 回到 ManagerAgent 的 `.planNotebook(planNotebook)`：**挂上它，主管就有了"把大任务自动拆成子任务"的能力**。
- 拆解算法在框架内部，你的代码只是"开启 + 配置"。

### 7.2 Hook —— 生命周期拦截器（看进度怎么打印的）
[planHook.java](manager_agent/src/main/java/com/imooc/managerAgent/hook/planHook.java)
```java
switch (event) {
    case PostReasoningEvent e -> { ...打印"第N轮思考"... }
    case PreActingEvent e ->    { ...打印"准备调用工具:xxx"... }
    case PostActingEvent e ->   { ...打印工具结果 + 计划状态... }
}
```
**你要明白的**：Hook 能在 ReAct 循环的**每一步前后插入逻辑**。这里用它来打印日志（你跑项目时控制台那些 ✅⏸▶️ 就是它打的）。**想看 Agent 内部在想什么、调了什么，就看这个类的输出。**

### 7.3 A2A + Nacos —— 跨进程调用
- 看 [NacosUtil.java](commons/src/main/java/com/imooc/commons/utils/NacosUtil.java)：连 Nacos 注册中心。
- **理解一句话就够**：专家 Agent 启动时把自己"登记"到 Nacos；主管要调用时去 Nacos"查找"。这样主管不用写死专家的地址(ip:port)，解耦。

**✅ 检查点**：你能分别说出这 3 个机制各解决什么问题（拆任务 / 看过程 / 跨进程找服务）。

## 第⑩站：跑起来 + 动手调试（最重要的一步）

**光看不练假把式。** 看完上面，一定要跑起来 + 打断点 + 改代码做实验。

### 启动顺序
```
1. 起 Nacos (localhost:8848)
2. 起 RouteMakingAgent (8082) 和 TripPlannerAgent (8085)
3. 最后起 ManagerAgent (8081)
```

### 发一个请求
```bash
curl -X POST http://localhost:8081/trip \
  -H "Content-Type: application/json;charset=utf-8" \
  -d '{"prompt":"推荐北京三天的景点和美食安排"}'
```

### 新手最该做的 3 个实验（理解远比读代码深）
1. **断点跟一遍**：在 `ManagerAgentController.tripPlan` 和 `RemoteAgentTool.callRouteMakingAgent` 打断点，单步跟着走一遍，亲眼看主管怎么调专家。
2. **盯日志看 ReAct 过程**：看控制台 `planHook` 打印的"第N轮思考""准备调用工具:xxx""计划状态"，理解 Agent 不是一步到位，而是"想—做—再想"循环。
3. **改 `@Tool` 的 description 看变化**：把 `callRouteMakingAgent` 的描述改模糊（比如改成"处理任务"），重启再问，看主管是不是就调错/不调了——亲身体会"描述决定路由"。

## 🗺️ 一页纸速记（贴墙上）

```

造Agent: AgentUtils.getReActAgentBuilder (大脑=DashScope)

3大机制: PlanNotebook(拆任务) / Hook(看过程) / A2A+Nacos(找服务)
嵌套链: Manager→(A2A)→RouteMaking→(MCP)→百度地图
        Manager→(A2A)→TripPlanner→(子Agent)→SuggestSight→(Skill)→脚本
```
