# AiTripPlan 代码逻辑详解

> 基于 **AgentScope 1.0.8 + Spring Boot 4 + JDK 21** 的多智能体（A2A）旅行规划系统。
> 本文逐层讲清楚：整体架构 → 核心概念 → 模块详解 → 完整调用链路 → 关键类解析。

---

## 一、一句话理解这个项目

> 用户发一句"帮我规划北京三日游"，**主管 Agent** 把任务拆成若干子任务，分别**委派**给两个专家 Agent（路线、行程）去做，最后汇总成完整方案返回。

这是一个典型的 **"主管 + 专家" 多智能体协作（Multi-Agent）** 架构。

---

## 二、整体架构

### 2.1 四个模块

| 模块 | 端口 | 角色 | 关键依赖 |
|------|------|------|----------|
| **commons** | - | 公共库（工具类、配置、数据结构），被其他三个模块依赖 | - |
| **manager_agent** | 8081 | **主管 Agent**，对外提供 REST 接口，负责任务拆解与调度 | Nacos、DashScope |
| **routeMaking_agent** | 8082 | **路线专家 Agent**（A2A 服务端） | Nacos、DashScope、百度地图 MCP |
| **tripPlanner_agent** | 8085 | **行程专家 Agent**（A2A 服务端） | Nacos、DashScope |

### 2.2 智能体层级（核心！）

这个项目有 **3 层嵌套的 Agent**，理解这个层级是看懂全局的关键：

```
用户 (HTTP POST /trip)
  │
  ▼
┌─────────────────────────────────────────────┐
│ ManagerAgent  主管 (8081)                      │
│  - 带 PlanNotebook（自动把任务拆成子任务）        │
│  - 工具 = 两个"远程Agent"（封装成可调用的工具）    │
└───────────────┬───────────────┬───────────────┘
                │ A2A 协议        │ A2A 协议
        (经 Nacos 发现)    (经 Nacos 发现)
                │               │
                ▼               ▼
┌───────────────────────┐  ┌──────────────────────────────┐
│ RouteMakingAgent       │  │ TripPlannerAgent  行程专家(8085)│
│  路线专家 (8082)         │  │  - 工具 = 子Agent              │
│  - 工具 = 百度地图 MCP    │  │         (SuggestSightAgent)   │
│  - 经 MCP/SSE 调地图API  │  └──────────────┬───────────────┘
└───────────────────────┘                 │ 子Agent当工具
                                           ▼
                              ┌────────────────────────────┐
                              │ SuggestSightAgent 景点推荐    │
                              │  - 挂载 SkillBox（技能）       │
                              │  - 能跑 Shell 脚本/读写文件     │
                              │  - 技能: Suggest-Sights /     │
                              │          Make-Table          │
                              └────────────────────────────┘
```

**三种不同的"工具化"方式**（AgentScope 的精髓）：
1. **远程 Agent 当工具**：Manager 把 RouteMaking / TripPlanner 当成工具调用（跨进程，A2A 协议）
2. **MCP 当工具**：RouteMaking 把百度地图 MCP Server 的能力当工具（跨服务，SSE 协议）
3. **子 Agent 当工具**：TripPlanner 把 SuggestSightAgent 当成本地工具（同进程）

## 三、AgentScope 核心概念速览

看代码前先了解这几个框架概念，否则会一头雾水：

| 概念 | 类 | 作用 | 本项目用在哪 |
|------|----|----|----|
| **ReActAgent** | `ReActAgent` | "推理(Reason)→行动(Act)"循环的智能体，会自己想、自己调工具 | 所有 Agent 的基类 |
| **PlanNotebook** | `PlanNotebook` | 让 Agent 把复杂任务**自动拆解成子任务**并跟踪状态 | ManagerAgent |
| **Toolkit** | `Toolkit` | 工具包，Agent 能调用的所有工具的集合 | 所有 Agent |
| **Tool** | `@Tool` 注解 | 把一个 Java 方法标记为 Agent 可调用的工具 | RemoteAgentTool |
| **MCP** | `McpClientWrapper` | Model Context Protocol，标准化的"外部工具服务"接入 | 百度地图 |
| **A2A** | `A2aAgent` | Agent-to-Agent 协议，让 Agent 跨进程互相调用 | Manager 调远程专家 |
| **Hook** | `Hook` | 生命周期拦截器，能在推理/调工具前后插入逻辑 | planHook（打印进度）|
| **SkillBox** | `SkillBox` | 技能箱，用 Markdown 定义技能 + 代码执行能力 | SuggestSightAgent |
| **DashScopeChatModel** | - | 阿里云通义千问大模型接入 | 所有 Agent 的"大脑" |

### ReAct 循环是什么？

每个 Agent 内部都是这样一个循环（直到任务完成）：

```
  ┌──────────────────────────────────────┐
  │  Reason 推理：我现在该做什么？           │  ← 调大模型思考
  │     ↓                                  │
  │  Act 行动：调用某个工具                  │  ← 执行工具
  │     ↓                                  │
  │  Observe 观察：工具返回了什么？           │  ← 拿到结果
  │     ↓                                  │
  │  (把结果喂回去，继续推理…)               │
  └──────────────────────────────────────┘
```

`planHook` 就是挂在这个循环上，把每一步的"思考""调哪个工具""工具结果""计划进度"打印出来。

## 四、模块详解

### 4.1 commons（公共模块）

所有共用的东西都在这。

| 文件 | 作用 |
|------|------|
| `conf/Properties.java` | 用 `@Value` 把 yml 里的配置（模型名、API Key、百度地址、LangFuse）读成 Bean |
| `utils/AgentUtils.java` | **核心工具类**：统一创建 ReActAgent（配模型、超时等） |
| `utils/PromptUtils.java` | 把用户的字符串 prompt 包装成框架要的 `Msg` 消息对象 |
| `utils/ToolUtils.java` | 创建/组装 Toolkit（支持三种工具来源） |
| `utils/NacosUtil.java` | 创建连 Nacos 的客户端（A2A 服务发现用） |
| `utils/LangFuseUtils.java` | 初始化 LangFuse 可观测（OpenTelemetry 上报链路） |
| `data/PromptSchema.java` | 请求体格式：`{ "prompt": "..." }` |
| `data/ResponseSchema.java` | 响应体格式：`{ "response": "..." }` |

**`AgentUtils.getReActAgentBuilder()` —— 全项目最关键的方法**

所有 Agent 都通过它来创建，统一了模型配置：

```java
ReActAgent.builder()
    .name(name).description(description)
    .model(DashScopeChatModel.builder()
        .apiKey(aliApiKey)        // 阿里云 Key
        .modelName(modelName)     // qwen 模型
        .stream(true)             // 流式
        .enableThinking(true)     // 思考模式
        .build())
    .checkRunning(true)           // 并发检查
    .toolExecutionConfig(...)     // 工具超时 120s，最多试 1 次
```

它返回的是 **Builder（半成品）**，各个 Agent 再往上加自己的 sysPrompt、工具包、计划等，最后 `.build()`。

### 4.2 manager_agent（主管，对外入口）

#### 调用入口：`ManagerAgentController`
```java
@RequestMapping(value = "/trip", method = POST)
public ResponseSchema tripPlan(@RequestBody PromptSchema input) {
    ResponseSchema response = managerAgent.run(input.getPrompt());
    return response;
}
```
接收 `{"prompt":"..."}`，调 `ManagerAgent.run()`，返回 `{"response":"..."}`。

#### 主管的"大脑"：`ManagerAgent`

**(1) 创建 Agent — `getManagerAgent()`**

```java
agent = agentUtils.getReActAgentBuilder("ManagerAgent", "...")
    .sysPrompt("你是一个旅游管理主管...先创建详细计划...按计划逐步执行")
    .planNotebook(planNotebook)              // ① 计划能力
    .hook(new planHook(planNotebook))        // ② 进度拦截器
    .toolkit(toolkit)                        // ③ 工具包(=两个远程Agent)
    .structuredOutputReminder(...)           // ④ 结构化输出
    .build();
```

四个关键配置：
- **① PlanNotebook**（来自 `TripPlan`）：让主管把"规划旅游"自动拆成 ≤5 个子任务，并跟踪每个的状态（待办/进行中/完成/放弃）。`needUserConfirm(false)` = 拆完不用人工确认，直接执行。
- **② planHook**：监听 ReAct 循环，打印思考、调用工具、计划进度（带 ⏸▶️✅❌ 图标）。
- **③ toolkit**：注册了 `RemoteAgentTool`，里面两个 `@Tool` 方法就是主管能"调用"的两个专家。
- **④ structuredOutputReminder**：要求模型按 `ResponseSchema` 输出结构化结果。
  > ⚠️ 实测：**planNotebook 计划流程下，本版本不会生成结构化输出**，详见第六节。

**(2) 运行 Agent — `run()`**

```java
public ResponseSchema run(String prompt) {
    PromptUtils promptUtils = new PromptUtils();
    Msg reply = agent
        .call(List.of(promptUtils.getPrompt(prompt)))  // 阻塞跑完整个流程
        .block();
    ResponseSchema result = new ResponseSchema();
    result.response = reply.getTextContent();          // 取最终回复文本
    return result;
}
```

`agent.call(...)` 会跑完整个 ReAct + 计划流程（期间会真的去调用远程专家 Agent），最后返回最终回复消息，取其文本作为结果。

#### 把远程 Agent 封装成工具：`RemoteAgentTool`

这是"主管能调度专家"的关键。两个方法，结构一样：

```java
@Tool(description = "擅长制定最优驾车路线")
public String callRouteMakingAgent(@ToolParam(...) String prompt) {
    // 1. 用 A2A 协议 + Nacos 发现，拿到远程的 RouteMakingAgent
    A2aAgent agent = A2aAgent.builder()
        .name("RouteMakingAgent")
        .agentCardResolver(new NacosAgentCardResolver(NacosUtil.getNacosClient()))
        .build();
    // 2. 组装 prompt，阻塞调用远程 Agent
    Msg userMsg = promptUtils.getPrompt("制定最优驾车路线：" + prompt + "...");
    Msg remoteAgentResponse = agent.call(userMsg).block();
    // 3. 返回远程 Agent 的文本结果
    return remoteAgentResponse.getTextContent();
}
```

**关键点**：
- `@Tool` 注解让这个方法变成主管 Agent 可调用的工具。主管的大模型看到工具描述"擅长制定最优驾车路线"，需要算路线时就会自动调它。
- `A2aAgent` + `NacosAgentCardResolver`：通过 **Nacos 注册中心**找到名为 `RouteMakingAgent` 的远程服务（跑在 8082），用 **A2A 协议**跨进程调用它。
- `callTripPlannerAgent` 同理，调的是 8085 的行程专家。

### 4.3 routeMaking_agent（路线专家）

#### `RouteMakingAgent`
```java
@Bean
public ReActAgent getRouteMakingAgent() {
    McpClientWrapper baiduMapMCP = mcp.getBaiduMapMCP();      // 创建百度MCP客户端
    McpClientWrapper mcpClient = mcp.initBaiduMapMCP(...);    // 初始化
    Toolkit toolkit = toolUtils.getToolkit(mcpClient);       // 把MCP工具装进工具包
    return agentUtils.getReActAgentBuilder("RouteMakingAgent", "...")
        .sysPrompt("你是一个驾车路线制定助手...制定性价比最优的路线")
        .toolkit(toolkit)
        .build();
}
```

- 用 `@Bean` 注册到 Spring 容器，**AgentScope 的 A2A starter 会自动把它注册到 Nacos**（见 `application.yml` 的 `agentscope.a2a.server.enabled: true`）。
- 工具来自**百度地图 MCP Server**。

#### `BaiduMapMCP`（MCP 客户端封装）
```java
McpClientWrapper baiduMapMCP = McpClientBuilder.create("BaiduMap-mcp")
    .sseTransport(properties.getBaiduMapAddr())  // SSE 方式连百度MCP (URL含AK)
    .timeout(Duration.ofSeconds(120))
    .buildAsync().block();
```
- 通过 **SSE** 连接百度地图官方 MCP Server（地址形如 `https://mcp.map.baidu.com/sse?ak=你的AK`）。
- `initBaiduMapMCP()` 用双重检查锁保证只初始化一次，初始化后能获取 MCP 服务端提供的地图工具列表（路线规划、距离计算等）。

### 4.4 tripPlanner_agent（行程专家）

这个模块最有意思，是 **2 层 Agent**。

#### `TripPlannerAgent`（行程专家本体）
```java
@Bean
public ReActAgent getTripPlannerAgent() {
    // 把"景点推荐子Agent"当成工具
    Toolkit toolkit = toolUtils.getToolkit(suggestSightAgent.getSuggestSightAgent());
    return agentUtils.getReActAgentBuilder("TripPlannerAgent", "...")
        .sysPrompt("你是一个旅游行程规划助手...景点、小吃、住宿...")
        .toolkit(toolkit)
        .build();
}
```
- 同样 `@Bean` 注册 → 自动进 Nacos。
- 它的工具不是 MCP，而是**另一个 Agent**（`SuggestSightAgent`）。`ToolUtils.getToolkit(ReActAgent)` 用 `registration().subAgent()` 把子 Agent 包装成工具。

#### `SuggestSightAgent`（景点推荐子 Agent）
```java
Toolkit toolkit = new Toolkit();
SkillBox skillBox = new SkillBox(toolkit);
skillBox.codeExecution()                    // 启用代码执行
    .workDir("./skill_file")
    .withShell(new ShellCommandTool(Set.of("bash")))  // 能跑 bash
    .withRead().withWrite()                 // 能读写文件
    .enable();

JarSkillRepositoryAdapter repo = new JarSkillRepositoryAdapter("skills");
AgentSkill SuggestSightsSkill = repo.getSkill("Suggest-Sights");  // 从jar读技能
skillBox.registerSkill(SuggestSightsSkill);

return agentUtils.getReActAgentBuilder("SuggestSightAgent", "...")
    .toolkit(toolkit).skillBox(skillBox).build();
```

- **SkillBox = 技能 + 代码执行能力**。
- **代码执行**：能在 `./skill_file` 目录跑 `bash` 命令、读写文件（⚠️ 需要环境有 bash）。
- **Skill（技能）**：从 `resources/skills/` 目录（打进 jar）以 Markdown 形式加载：
  - `Suggest-Sights`：定义了"如何推荐景点/美食/住宿"的能力说明（见 `SKILL.md`）。
  - `Make-Table`：定义了"把内容写成表格并调 `scripts/table.sh` 生成文件"的能力。

> **技能（Skill）本质**：用 Markdown 写的"操作手册"，告诉大模型遇到某类任务该怎么一步步做、该调哪个脚本。模型读懂手册后照着执行。

## 五、完整调用链路（端到端）

以"帮我规划北京到天津两日游，含路线+景点"为例，完整走一遍：

```
①  用户 POST http://localhost:8081/trip  {"prompt":"..."}
        │
②  ManagerAgentController.tripPlan() → ManagerAgent.run()
        │
③  主管 ReActAgent 启动，PlanNotebook 把任务拆成子任务，例如：
        ⏸ [0] 制定北京→天津驾车路线
        ⏸ [1] 规划天津景点/美食/住宿
        ⏸ [2] 汇总完整方案
        │ (planHook 实时打印拆解过程)
        │
④  主管按计划执行 —— 推理后决定调用工具 callRouteMakingAgent
        │
        ├─▶ RemoteAgentTool.callRouteMakingAgent()
        │       │ A2aAgent + Nacos 发现 → 找到 8082 的路线专家
        │       ▼
        │   RouteMakingAgent (8082) 收到任务
        │       │ ReAct 循环：调用百度地图 MCP 工具
        │       ▼
        │   百度地图 MCP Server (SSE) → 返回路线/里程/高速费
        │       │
        │   ◀── 路线方案文本返回给主管
        │
⑤  主管继续执行 —— 调用工具 callTripPlannerAgent
        │
        ├─▶ RemoteAgentTool.callTripPlannerAgent()
        │       │ A2aAgent + Nacos 发现 → 找到 8085 的行程专家
        │       ▼
        │   TripPlannerAgent (8085) 收到任务
        │       │ ReAct 循环：调用子Agent工具 SuggestSightAgent
        │       ▼
        │   SuggestSightAgent：读 Skill 手册 → 推荐景点/美食/住宿
        │                      → (可选)跑 table.sh 生成表格
        │       │
        │   ◀── 行程方案文本返回给主管
        │
⑥  主管把各专家结果汇总，PlanNotebook 全部子任务标记 ✅
        │
⑦  agent.call().block() 返回最终回复消息
        │
⑧  ManagerAgent.run() 取 reply.getTextContent() 塞进 ResponseSchema
        │
⑨  返回 {"response":"...完整行程..."} 给用户
```

**贯穿全程的可观测**：每个 Agent 的推理/工具调用都通过 OpenTelemetry 上报到 **LangFuse**（如已配置），可在 LangFuse 看到完整的调用链路。

## 六、一个重要的"坑"：结构化输出 + 计划流程不兼容

### 现象
`ManagerAgent.run()` 原本这样取结果：
```java
stream.blockLast().getMessage().getStructuredData(ResponseSchema.class);
```
运行时报错：`No structured output in message metadata. Key '_structured_output' not found.`

### 根因
- 主管 Agent 配了 `PlanNotebook`，它以 `finish_subtask`（完成子任务）方式收尾。
- **AgentScope 1.0.8 在计划流程下不会在终结消息上生成 `_structured_output` 元数据**。
- 无论用 `stream + blockLast` 还是官方的 `call(msg, Schema.class)`，结果一样——都拿不到结构化数据。

### 解决
注意 `ResponseSchema` 其实只有一个字段：
```java
public class ResponseSchema { public String response; }
```
所谓"结构化输出"就是把答案套进 `response` 字段而已，没有真正的嵌套结构。所以直接取最终回复文本即可，**结果完全等价**：
```java
Msg reply = agent.call(List.of(promptUtils.getPrompt(prompt))).block();
result.response = reply.getTextContent();
```

> 这不是"降级"，而是认清：对单字段 schema，文本本身就是答案，结构化输出机制纯属多余且在计划流程下失效。

## 八、配置与启动

### 配置（根目录 `.env`）
```properties
ALIBABA_DASHCOPE_KEY=sk-xxx                          # 阿里云大模型Key（注意拼写DASHCOPE）
BAIDU_MAP_ADDR=https://mcp.map.baidu.com/sse?ak=xxx  # 百度地图MCP地址(含AK)
MODEL_NAME=qwen3.7-max                               # 模型名
LANGFUSE_ADDR / SECRET_KEY / PUBLIC_KEY              # LangFuse可观测(可选)
```
- manager / tripPlanner 的 yml 用 `${MODEL_NAME}` 占位符从 `.env` 读取。
- routeMaking 的 yml 是**写死**的值（与另两个不一致，注意）。

### 启动顺序（不能乱）
```
1. Nacos (localhost:8848)              注册中心，必须先起
2. (路线功能需要) 百度地图 MCP 可达
3. RouteMakingAgent (8082)             ┐ 启动后注册到 Nacos
4. TripPlannerAgent (8085)             ┘
5. ManagerAgent (8081)                 最后起，从 Nacos 发现专家
```

### 测试
```bash
curl -X POST http://localhost:8081/trip \
  -H "Content-Type: application/json;charset=utf-8" \
  -d '{"prompt":"推荐北京三天的景点和美食安排"}'
```

## 九、设计亮点总结

1. **分层多智能体**：主管负责"调度大局"，专家负责"垂直能力"，职责清晰、可独立扩展。
2. **三种工具化手段**：远程Agent(A2A) / MCP / 子Agent，覆盖了 AgentScope 工具体系的主要玩法。
3. **A2A + Nacos 解耦**：专家 Agent 独立部署、独立扩缩容，主管通过注册中心动态发现，不硬编码地址。
