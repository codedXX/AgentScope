# AiTripPlan-AgentScope 项目代码逻辑全面分析

## 1. 项目概览
该项目是一个基于 **Spring Boot** 和 **AgentScope** 框架开发的分布式多智能体（Multi-Agent）旅游规划系统。
系统采用了微服务架构，各个 Agent 独立作为 Spring Boot 模块运行，通过 **Nacos** 进行服务注册与发现，利用 AgentScope 的 **A2A（Agent-to-Agent）协议**进行远程通信协同工作。系统还接入了 **Langfuse** 进行大模型交互的链路追踪，底座大模型采用的是**通义千问（DashScope API）**。

---

## 2. 核心模块与架构
项目分为四大核心模块，各司其职：
* **`commons`（公共模块）**：存放全局配置、实体类（`PromptSchema`、`ResponseSchema`）以及共用工具类（`AgentUtils`、统一的大模型配置等）。
* **`manager_agent`（主管模块）**：系统的中枢大脑，负责接收用户请求、制定执行计划、并将子任务分发给各个专家智能体。
* **`routeMaking_agent`（路线制定模块）**：领域专家，负责通过百度地图接口获取真实数据，制定最优自驾路线。
* **`tripPlanner_agent`（行程规划模块）**：领域专家，负责详细的旅游景点、食宿推荐，内部采用了嵌套子智能体和外部技能（Skills）挂载。

---

## 3. 各模块代码逻辑深入拆解

### 3.1 主管 Agent (`ManagerAgent`)
主管 Agent 是整个规划流程的发起者和控制者。
* **交互入口 (`ManagerAgentController`)**：暴露了 `/trip` 的 RESTful API 接口，负责接收前端（或用户）传来的旅游意图（Prompt），并返回结构化的响应数据。
* **任务分解与调度 (`ManagerAgent.java`)**：
  - 基于 `ReActAgent` 模式，被赋予了“旅游管理主管”的系统设定。
  - **核心机制 `PlanNotebook`**：代码中使用了 `PlanNotebook` 对复杂的旅游需求进行自主拆解。它能把一个宏大的任务（如“从深圳到惠州3日游”）拆解为有序的多个执行步骤，并全程跟踪每个步骤的完成状态。
  - **拦截器（`planHook`）**：在计划的执行生命周期中进行拦截，以便记录或干预。
* **远程调用封装 (`RemoteAgentTool.java`)**：
  - 将其他独立模块的 Agent 封装成“大模型的工具”（Tool）。
  - 提供 `callRouteMakingAgent` 和 `callTripPlannerAgent` 方法。当大模型决定使用这些工具时，该类通过 `NacosAgentCardResolver` 从 Nacos 注册中心查找远程 Agent 的地址，并利用 A2A 协议 (`agent.call().block()`) 发起阻塞式的 RPC 调用。

### 3.2 路线制定 Agent (`routeMaking_agent`)
专门解决交通（怎么去、走哪条路、多少钱）的问题。
* **代理定义 (`RouteMakingAgent.java`)**：系统设定其为“驾车路线制定助手”，目标是找出性价比最优路线。
* **MCP 架构集成 (`BaiduMapMCP.java`)**：
  - 此处应用了先进的 **MCP (Model Context Protocol)** 架构。它并不是手写一堆 HTTP 请求去调百度地图，而是通过 `McpClientBuilder` 异步构建了一个 MCP 客户端，通过 SSE (Server-Sent Events) 与独立的百度地图 MCP Server 建立通信。
  - 将这个 `McpClientWrapper` 包装成 `Toolkit` 提供给大模型，让大模型能直接拥有调用百度地图查询里程、高速收费等真实世界数据的能力。

### 3.3 行程规划 Agent (`tripPlanner_agent`)
解决“去哪玩、住哪、吃啥”的问题，此模块采用了**父子智能体嵌套**的高阶玩法。
* **主规划师 (`TripPlannerAgent.java`)**：
  - 扮演“旅游行程规划助手”统筹全局安排。
  - 它不直接干脏活，而是把专注景点的 `SuggestSightAgent` 包装成工具包 (`toolkit`) 挂载给自己。同时，该 Agent 会在应用启动时自动注册到 Nacos，供 `ManagerAgent` 调用。
* **景点推荐子代理 (`SuggestSightAgent.java`)**：
  - 它是行程规划师背后的“景点领域专家”。
  - **代码执行沙箱**：它配置了强大的 `skillBox.codeExecution()`，启用了 Shell 命令 (`ShellCommandTool`)、文件读写能力，工作目录被限定在 `./skill_file`。这意味着大模型可以通过写代码并执行来完成复杂的数据处理。
  - **本地技能库**：通过 `JarSkillRepositoryAdapter` 从本地库中加载了特定的技能（如 `Suggest-Sights` 推荐技能和 `Make-Table` 制表技能），让它在输出特定格式的行程表时更加专业和稳定。

---

## 4. 底层支撑机制

* **大模型统一配置 (`AgentUtils.java`)**：
  项目中的所有 Agent 都在这里统一构建。统一使用了阿里云通义千问的大语言模型配置，默认开启了**思考模式** (`enableThinking(true)`) 增强推理能力，开启了**流式响应** (`stream(true)`)，并将每个工具的执行超时时间严格限制为 120 秒。
* **结构化数据约束 (`ResponseSchema.class`)**：
  在 `ManagerAgent` 实例化时，配置了 `structuredOutputReminder(StructuredOutputReminder.PROMPT)`。当 Agent 执行完毕后，利用泛型反序列化，确保最终吐出给用户的数据是严格遵循 `ResponseSchema` 结构的对象，方便前端直接使用，避免了大模型乱讲话的问题。

## 5. 完整执行流总结
当你向系统发送一句类似 **“帮我制定元旦深圳到惠州3日游自驾游计划”** 的请求时，系统内部发生了如下连锁反应：

1. **接单**：请求到达 `ManagerAgentController`。
2. **拆解**：`ManagerAgent` 开始思考，利用 `PlanNotebook` 制定执行计划表（第一步查路线，第二步排行程）。
3. **查路线**：主管大模型决定执行第一步，调用 `RemoteAgentTool.callRouteMakingAgent`。系统通过 Nacos 找到 `routeMaking_agent` 服务并发起远程调用。
4. **获取地图数据**：路线专家接到任务，向大模型后台请求，大模型判定需要调用外部接口，触发 `BaiduMapMCP` 向百度地图 MCP 服务端要数据。拿到里程和路费后返回给主管。
5. **排行程**：主管大模型进入下一步，调用 `RemoteAgentTool.callTripPlannerAgent` 唤醒远端的行程专家。
6. **景点研究**：行程专家遇到景点难题，调用挂载在自己身上的 `SuggestSightAgent`。小专家利用其内置的 Shell 和读取文件等 `Skill` 进行分析制表，层层上报。
7. **交付**：所有环节执行完毕，`ManagerAgent` 将收集到的海量信息汇总，通过结构化转换，向你返回最终完美的旅游计划。
